\subsubsection{MetricsSection Component}
\label{sec:metricssection}
 
\paragraph{Purpose}
This component gives users a quick, clear look at how their workflows have
performed over the last seven days. It displays a row of four distinct
summary cards -- Total, Failed, and Successful executions, alongside an
overall Success Rate -- followed by a visual trend chart.
 
\begin{lstlisting}[language=TypeScript]
<Row className="metrics-row">
  {METRICS_CONFIG.map((config, index) => {
    const { key, ...metricProps } = config;
 
    return (
      <Col key={key} md={6} lg={3} className="mb-3">
        <MetricCard
          {...metricProps}
          index={index}
          isLoading={isLoading}
          onMetricClick={onMetricClick}
          value={
            analytics?.[key] != null
              ? metricProps.format(analytics[key])
              : "--"
          }
        />
      </Col>
    );
  })}
</Row>
\end{lstlisting}
 
\paragraph{Location}
\texttt{src/components/MetricsSection.jsx}
 
\paragraph{Props}
\begin{itemize}
    \item \texttt{setAuth} (Function): Updates user login or profile
    details if something expires or errors out during background data
    collection.
    \item \texttt{onMetricClick} (Function): A callback that triggers when
    a user clicks on a card. It takes them directly to a pre-filtered list
    of executions (for example, showing only failed runs).
    \item \texttt{onAnalyticsLoaded} (Function): An optional helper that
    sends the freshly fetched data back up to parent components that might
    need it.
\end{itemize}
 
\paragraph{Internal Components}
\begin{itemize}
    \item \textbf{MetricCard}: The individual card layout for each metric.
    It manages its own loading spinners, entry animations, text styling,
    and an animated count-up effect when numbers load.
\end{itemize}
 
\paragraph{Behavior \& Logical Flows}
\begin{enumerate}
    \item \textbf{Automatic Data Fetching}: As soon as the component
    appears on screen, it automatically contacts
    \texttt{ExecutionService.getAnalytics} to grab data from the past week.
    \item \textbf{Staggered Card Entry}: Each card waits for a brief,
    incremental delay before fading into view. This prevents everything
    from popping in abruptly.
    \item \textbf{The Count-Up Animation}: When the data finishes loading,
    numbers do not just snap into place. Instead, a built-in timer splits
    the target value into 30 small chunks, rolling the counter up over a
    smooth 900-millisecond window. It also preserves percentage symbols or
    text suffixes.
    \item \textbf{Interactive Filtering}: Cards that handle status filters
    (like ``Failed'' or ``Success'') become interactive. Hovering over
    them reveals a pointer cursor, and clicking them updates the
    application's view state to display those specific execution records.
\end{enumerate}

\subsubsection{Overview Page in Restricted Mode}
\label{sec:overview-restricted-mode}
 
To prevent unauthorized access to workflow management features, the
Overview page enters a restricted mode whenever the user's email address
is not verified. This state is triggered after a user changes their
email address, requiring verification before sensitive operations become
available again.
 
In this mode, the page remains accessible to preserve the application's
layout and provide visual context, but all protected functionality is
disabled. Instead of displaying an empty page, the interface presents
placeholder content and a verification prompt, allowing users to
understand the available features while clearly indicating that
verification is required.
 
\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{images/overview-page/disabled-overview.png}
    \caption{The Overview Page in restricted mode, showing blurred metrics, a disabled Workflows tab, and the email verification prompt.}
    \label{fig:overview-restricted-mode}
\end{figure}
 
The restricted mode includes the following behavior:
\begin{itemize}
    \item The Create Workflow button is disabled and displays a tooltip
    indicating that email verification is required.
    \item The Metrics section remains visible but is covered by a blur
    overlay, preventing interaction with workflow statistics.
    \item The Credentials and Executions tabs are disabled and cannot be
    selected until the email is verified.
    \item The Workflows tab remains accessible but displays placeholder
    workflow cards instead of the user's actual workflows.
    \item A centered overlay containing a lock icon, an explanatory
    message, and a Verify Email button is displayed over the tab
    content.
    \item Selecting Verify Email resends the verification email, when
    necessary, and redirects the user to the email verification page to
    complete the verification process.
\end{itemize}
 
Rather than hiding the Overview page completely, the system adopts a
progressive restriction approach. Users can still navigate the interface
and become familiar with the available functionality, while all
operations that require a verified identity remain securely disabled.
This approach improves the overall user experience by clearly
communicating why certain features are unavailable and guiding users
toward the verification process required to unlock them.
