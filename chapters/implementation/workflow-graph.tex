\section{Workflow Graph Construction, Validation}

\subsection{Data Model}

The type system is defined in \texttt{workflow.type.ts}. \textbf{\texttt{INode}}
represents a single workflow node: \texttt{id}, \texttt{type} (registry key),
\texttt{label}, \texttt{projectId}, \texttt{executionId} (assigned at runtime
by the worker), an \texttt{inputSource} policy
(\texttt{'self'|'parent'|'combined'|'any'}), \texttt{inputs}/\texttt{parentOutputs}
arrays, \texttt{parameters}, optional \texttt{credentials}, and resolved
\texttt{children: INode[]}. \textbf{\texttt{IConnection}} is a minimal directed
edge \texttt{\{source, target\}}, and \textbf{\texttt{IWorkflowGraph}} is
\texttt{\{nodes: INode[]; connections: IConnection[]\}}. A fourth interface,\\
\textbf{\texttt{IWorkflowExecutionContext}}, bundles a built
\texttt{WorkflowGraph} with \texttt{projectId} and is passed from the
controller to the runner.

\subsection{The \texttt{WorkflowGraph} Class}

Wraps an \texttt{IWorkflowGraph} with an \textit{adjacency list}
(\texttt{Map<string,string[]>}) for $O(1)$ neighbour lookup, kept in sync
whenever \texttt{addConnection} is called. Both \texttt{addNode} and
\texttt{addConnection} throw an \texttt{AppError} on duplicates, preventing
malformed payloads from silently corrupting the graph.

\textbf{Finding trigger nodes.} A trigger node is defined structurally as any
node with zero incoming edges. \texttt{findTriggerNodes()} builds the set of
every node that appears as an edge \textit{target}; any node absent from that
set is a trigger candidate. Actual trigger-type validity is confirmed later
against the node registry.

\textbf{Decomposing into connected components.} A canvas may hold multiple
independent sub-workflows. \texttt{findConnectedComponents()} runs BFS over
the \textit{undirected} version of the graph (forward and reverse adjacency),
so components are found correctly regardless of edge direction --- a
directed-only BFS from a trigger could miss nodes upstream of a cycle. Only
connections whose \textit{both} endpoints belong to a component are copied
into it.

\subsection{Building the Graph from the HTTP Request}

\texttt{getWorkflowGraph()} (in \texttt{workflow-pre-execution.validations.ts})
turns the raw frontend payload into a typed \texttt{WorkflowGraph}. The
frontend (React Flow) sends \texttt{nodes[]} and \texttt{connections[]}, with
each node's semantic type nested at \texttt{data.nodeName} rather than
top-level.

\begin{enumerate}
  \item Shape-validate \texttt{nodes} (array, string \texttt{id}s) and
        \texttt{connections} (array with string \texttt{source}/\texttt{target}/
        \texttt{type}) --- throw \texttt{AppError(400)} on failure.
  \item Map each raw node to a typed \texttt{INode}, unwrapping
        \texttt{data.nodeName} $\rightarrow$ \texttt{type}. \texttt{executionId}
        is left \texttt{''} (assigned later by the worker); \texttt{children}
        is left \texttt{[]} (resolved recursively at deployment time via
        \texttt{buildChildren}, so the full execution tree travels with each
        job without further lookups).
  \item Populate a new \texttt{WorkflowGraph} via \texttt{addNode}/
        \texttt{addConnection}.
\end{enumerate}

\subsection{Pre-Execution Validation Pipeline}

\texttt{validateAllWorkflows()} decomposes the graph into connected components
and runs each through three validators, returning
\texttt{\{validWorkflows, invalidWorkflows\}} (the latter paired with error
messages). Any invalid component causes the whole deployment request to be
rejected with a structured error payload.

\textbf{1. Trigger node validation} (\texttt{validateTriggerNodes}, runs
first) enforces: (a) the component must contain $\geq 1$ node; (b) every
zero-in-degree node must be registry-declared as \texttt{triggerNode: true};
(c) no node \textit{with} incoming edges may be a trigger type --- triggers
are only valid at a component's root.

\textbf{2. Node registry validation} (\texttt{checkAllNodesValidation}, runs
second) confirms every \textit{non-trigger} node's \texttt{type} exists in the
\texttt{NodeRegistry} (triggers are already checked in step 1). Unregistered
nodes are collected into a single list so the user sees every offending node
at once rather than one at a time.

\textbf{3. Cycle detection} (\texttt{checkCycles}, runs last, and only if
steps 1--2 passed) uses a classic three-colour DFS via a \texttt{VisitState}
enum (\texttt{UNVISITED / VISITING / VISITED}). A node found in
\texttt{VISITING} state while traversing its descendants indicates a back
edge $\rightarrow$ cycle. This check is deliberately gated behind the
structural checks: a DFS over a graph with unregistered nodes or no valid
trigger cannot be trusted to reflect the intended structure, and an
undetected cycle would later cause \texttt{buildChildren} to recurse
infinitely, flooding the job queue.

  2 & \texttt{checkAllNodesValidation} &

  \bottomrule
\end{tabular}

Trigger and registry errors are accumulated independently and both reported
if both fail; the cycle check only runs when registry validation passed
